# LangGraph를 Elixir로 구현할 때의 이득 — Python 구현과의 상세 비교

> ElGraph가 Python LangGraph 대비 무엇을 구조적으로 더 잘할 수 있고, 무엇을 포기하는지의 정직한 비교.
> 결론 요약: **오케스트레이션 런타임 품질은 Elixir의 구조적 우위, 모델/생태계 인접 작업은 Python의 우위.** 에이전트 오케스트레이터는 본질적으로 "수많은 동시 I/O 대기 + 상태 관리 + 장애 복구" 문제이므로 런타임 우위가 코어 가치에 직결된다.

작성: 2026-06-11

---

## 1. 동시성 / 실행 모델

| 항목 | Python (LangGraph) | Elixir (ElGraph) | 이득 |
|---|---|---|---|
| 동시 실행 단위 | asyncio 태스크(단일 스레드 이벤트 루프) 또는 스레드(GIL 제약) | BEAM 경량 프로세스 (초기 힙 ~수 KB, 수백만 개 가능) | 그래프 호출당 프로세스 1개를 그냥 띄움. 동시 실행 수만 건이 설계 변경 없이 동작 |
| CPU 병렬성 | GIL로 단일 코어 직렬화 (3.13 free-threading은 아직 생태계 미성숙) | 모든 코어를 스케줄러가 자동 활용 (per-core run queue) | reducer 병합·직렬화·라우팅 같은 CPU 작업이 코어 수만큼 병렬 |
| 블로킹 호출의 영향 | 동기 호출 하나가 이벤트 루프 전체를 멈춤 (`async`/`sync` 이중 API 강제 — LangGraph는 `invoke`/`ainvoke` 양쪽 유지) | 선점형 스케줄링 — 한 프로세스의 블로킹이 다른 프로세스에 영향 없음 | **API가 하나로 통일**. 색깔 있는 함수(colored functions) 문제 자체가 없음 |
| 공정성 | 협조적 양보(`await`) 필수. 양보 안 하는 태스크가 전체 지연 | reduction 카운트 기반 선점 — 폭주 노드가 있어도 다른 실행은 공정하게 진행 | 멀티테넌트 환경에서 한 사용자의 무거운 그래프가 다른 사용자를 굶기지 않음 |
| 병렬 fan-out (Send/map-reduce) | asyncio.gather — 같은 루프 안에서 인터리빙 | Task.Supervisor로 실제 병렬 + 개별 장애 격리 | 브랜치 하나의 크래시가 다른 브랜치에 전파되지 않음 |

## 2. 상태 관리 / 정확성

| 항목 | Python | Elixir | 이득 |
|---|---|---|---|
| 상태 가변성 | dict는 가변 — 노드가 받은 상태를 in-place 수정하면 조용한 버그 (LangGraph 문서가 "복사해서 쓰라"고 경고하는 수준) | 모든 데이터 불변 — 노드가 상태를 오염시키는 것이 **언어적으로 불가능** | 채널/reducer 모델의 핵심 불변식("쓰기는 reducer로만 병합")을 런타임이 보장 |
| 병렬 브랜치 간 간섭 | 같은 객체 참조를 공유 — aliasing 버그 가능 | 프로세스 간 데이터 공유 없음(복사 전달) | 병렬 노드의 데이터 레이스가 원천 불가능 |
| 패턴 매칭 | match 문(3.10+)은 제한적 | 함수 헤드 패턴 매칭 + 가드 | 노드 라우팅·메시지 타입 분기 코드가 짧고 망라성 검증이 쉬움 |
| 큰 바이너리(LLM 응답) | 문자열 복사 빈번 | 64바이트 초과 바이너리는 refc로 프로세스 간 공유 (복사 없음) | 긴 컨텍스트를 병렬 브랜치에 전달할 때 메모리 이득 |

## 3. 장애 격리 / 복구 (에이전트 프레임워크의 핵심 차별점)

| 항목 | Python | Elixir | 이득 |
|---|---|---|---|
| 노드 크래시 | try/except를 모든 경계에 직접 작성. 놓치면 전체 프로세스로 전파 | 프로세스 격리 — 노드 Task 크래시는 해당 실행만 영향, 러너가 `:DOWN`으로 수신 | "한 에이전트의 죽음이 다른 에이전트를 못 죽인다"가 기본값 |
| 복구 전략 | 애플리케이션 코드 + 외부 인프라(K8s 재시작, Celery 재시도)로 해결 | Supervisor 트리 — 재시작 전략(one_for_one 등)이 언어 표준 | 장수명 에이전트의 자기 치유가 프레임워크 바깥 인프라 없이 성립 |
| 장수명 에이전트 | 프로세스 죽으면 전부 죽음 → 사실상 상태를 전부 외부 DB에 두고 stateless 서버로 우회 | crash-only 설계 + 체크포인트 복구 + 슈퍼바이저 재시작이 한 몸 | "항상 살아있는 에이전트"를 자연스럽게 모델링 |
| 좀비/리소스 누수 | asyncio 태스크 취소는 협조적이며 누수 사례 빈번 | 프로세스 종료 시 링크/모니터로 연쇄 정리, 소유 리소스(소켓 등) 자동 해제 | 취소 시맨틱(§SPEC 3.9)이 신뢰 가능 |
| 런타임 관측 | 외부 APM 의존 | `:observer`, `:sys.get_state/1`, 프로세스별 메일박스/메모리 실시간 조회 | 운영 중 "이 에이전트 지금 뭐하나"를 REPL에서 즉시 확인 |

| 항목 | Python | Elixir | 이득 |

## 5. 분산 / 스케일아웃

| 항목 | Python | Elixir | 이득 |

## 7. Python이 우위인 것 (정직한 비용)

| 항목 | 내용 | ElGraph의 대응 |
|---|---|---|
| LLM/ML 생태계 | 프로바이더 SDK, 토크나이저, 평가 도구, 벡터 DB 클라이언트가 Python 우선 출시 | LLM API는 결국 HTTP+SSE — Req로 직접 구현(SPEC §4). MCP 어댑터로 툴 생태계 흡수 |
| 임베딩/로컬 모델 | PyTorch 생태계 독점적 | Nx/Bumblebee가 있으나 성숙도 열세. 임베딩은 API 호출로 우회 가능 |
| 커뮤니티/인력 | LangGraph 사용자 풀, 튜토리얼, 채용 풀 모두 압도적 | 니치 전략: Elixir 커뮤니티에는 경쟁자가 사실상 없음(SPEC §1) |
| 레퍼런스 구현 | LangGraph가 이미 검증한 설계(channels, pending writes, Send)를 베낄 수 있음 | 이것은 오히려 이득 — 설계 리스크를 LangGraph가 이미 지불 |
| 타입 도구 | mypy/pyright 성숙 | Dialyzer는 점진적·보수적. (Elixir 공식 set-theoretic 타입이 진행 중이나 아직 부분적) |
| 토크나이저 정밀 카운팅 | tiktoken 등 즉시 사용 | API의 usage 응답으로 우회하거나 NIF 바인딩 필요 |

## 8. 종합

LangGraph가 Python 위에서 **라이브러리로 재구현해야 했던 것들** — 체크포인트 기반 내구 실행, 병렬 브랜치 격리, 백그라운드 실행 관리, 스트리밍 버스, 분산 워커 — 의 상당 부분이 BEAM에서는 **런타임 기본 제공**이다. 즉 Elixir 구현은 "같은 기능을 더 적은 코드와 더 적은 인프라로, 더 강한 보장과 함께" 제공할 수 있다.

반대로 모델 인접 작업(임베딩, 토크나이저, 평가 생태계)은 Python이 우위이며, ElGraph는 이를 **HTTP API와 MCP로 우회**하는 전략을 취한다(SPEC §4). 오케스트레이터가 모델을 직접 실행할 일은 없으므로 이 우회는 구조적으로 타당하다.

| 판단 기준 | 결론 |
|---|---|
| 동시 실행 1만+ 에이전트, 장수명 에이전트, 실시간 UI, 자기 치유 | **Elixir 압도적 우위** |
| 빠른 프로토타이핑, ML 실험, 로컬 모델 직접 구동 | Python 우위 |
| 에이전트 **오케스트레이션 프레임워크**라는 본 프로젝트의 목적 | Elixir의 우위 영역과 정확히 겹침 — 구현 타당 |
